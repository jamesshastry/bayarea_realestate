## Part 1 — First Impressions & Onboarding (UserOnboard)

### Findings

- **Anonymous-first onboarding** (operating principle #7) is the right call for FTHB; every competitor traps users behind email-gate.
- **No defined "first-3-inputs" empty state** — a new visitor sees blank affordability fields with nothing to anchor on.
- **Tooltips for jargon** (DTI, jumbo, PMI, Mello-Roos, Prop 13) exist (F-EDU-02) but spec doesn't say *when* they're triggered. Hover-only fails on mobile.
- **Cost-of-Waiting calculator** isn't surfaced to users who only used affordability — risk of the timing surface being undiscovered.

### New requirements added to `requirements.md`

- **F-AFF-10** Affordability widget MUST prefill with a median Bay Area FTHB profile (e.g., $300K income, $150K down) on first visit, with a clear "this is an example — change it" banner. localStorage overrides the example after first user input.
- **F-EDU-04** Every defined glossary term in the UI MUST be a tappable badge that opens an inline definition card on tap (not just hover) — required for mobile parity.
- **F-AFF-11** After 30 seconds on an area page, a non-modal banner SHOULD prompt the user toward Cost-of-Waiting if they haven't yet visited it.
- **F-SCN-08** After the 2nd or 3rd anonymous interaction with affordability inputs, a non-blocking "save these inputs" prompt SHOULD appear (account creation is optional; can also save by emailing the link).

### Score

| Surface | Score |
|---|---|
| Spec | 7/10 |
| Current MVP | 4/10 |

## Part 2 — Navigation & Information Architecture (Mobbin)

### Findings

- URL-as-state (design §7.3) is correct — bookmarkable comparisons are a real moat.
- Metro namespacing (F-MM-02) works.
- **No top-level nav structure defined** in spec.
- **No global search UI** despite `/v1/areas/search` existing in the API.
- **No breadcrumb requirement** — wayfinding gets ambiguous as nesting grows.
- **Comparison multi-select** is fiddly without a sticky drawer pattern.

### New requirements added to `requirements.md`

- **F-NAV-01** Primary nav MUST be exactly: `Areas | Timing | Compare | Map | Saved | Learn` (6 items; fits desktop tab bar and mobile bottom nav). Cities/schools/zips/neighborhoods are reached *through* `Areas`, not as top-level items.
- **F-NAV-02** A global Cmd+K command palette MUST be available on every page, supporting search by area name, school, ZIP, glossary term. Built on shadcn `<Command>` (cmdk).
- **F-NAV-03** Every page below `/[metro]` MUST render a breadcrumb (e.g., `Bay Area › Cities › Fremont › Schools › Foothill HS`).
- **F-CMP-05** "Add to compare" button MUST exist on every area card; clicking populates a sticky bottom drawer showing the active comparison set with quick-remove.

### Score

| Surface | Score |
|---|---|
| Spec | 6/10 |
| Current MVP | 5/10 |

## Part 3 — Core Task Flows (Built for Mars step-by-step)

### Affordability flow (analog of "device control") — spec'd

- Spec hits the right beats: 200ms recompute (NF-PRF-02), inline Mello-Roos surfacing, "show the math" expandable cards.
- Risks: monthly breakdown spec'd as "show breakdown" — could be misread as a modal/page nav. RSU input split (F-AFF-01) adds friction for non-tech users.

### Cost-of-Waiting flow (analog of "automation") — spec'd

- 3×3 scenario grid is right-sized. "Break-even rate drop" is the killer metric.
- Risks: unscanable on mobile without a layout-collapse rule. No way to *save a scenario from this flow* — users tinker, close tab, lose it.

### Saved-area / alert flow (analog of "notification management") — Phase 4

- One-click save creates subscription + sensible defaults (no "set up alerts" step).
- Risks: 4-checkbox modal is too much. Snooze (F-RT-09) exists in data model but no UI flow.

### Comparison flow — Phase 3

- Drawer pattern is conventional and works. Cell-level "show the math" preserves operating principle #1.
- Risks: 4 areas × 12 metrics is a horizontal-scroll nightmare on mobile. No row-pin / row-hide for power users.

### New requirements added to `requirements.md`

- **F-AFF-12** The monthly cost breakdown (per F-AFF-04) MUST be inline-expandable on the same page — never a modal or separate page navigation.
- **F-AFF-13** Income input defaults to a single "household income" field. Splitting into base / bonus / RSU MUST be a progressive opt-in toggle, not always visible.
- **F-TIM-10** Cost-of-Waiting page MUST have a "Save this scenario" button that writes to localStorage anonymously and prompts auth on second visit.
- **F-TIM-11** On viewports < 768px, the Cost-of-Waiting 3×3 grid MUST collapse to a vertical card list with the base case rendered first.
- **F-RT-12** Alert subscription UI MUST default to a curated bundle ("watch for material market changes"); per-signal customization is behind a "Custom thresholds" disclosure.
- **F-RT-13** Every dispatched alert (in-app or email) MUST include "Mute 7d" / "Mute 30d" / "Edit thresholds" inline actions.
- **F-CMP-06** On viewports < 768px, the comparison matrix MUST collapse to one accordion per area with metrics inline.
- **F-CMP-07** Comparison view MUST support column-group show/hide (price, schools, commute, risk) for power-user focus.

### Score

| Surface | Score |
|---|---|
| Spec, taken as a whole | 7/10 |

## Part 4 — Visual Design & Micro-interactions

### Findings

- shadcn + Radix + Tailwind + OKLCH + Tremor + Visx + Mapbox stack (design §10.7) is well-chosen.
- Component inventory covers every recurring spec pattern.
- Tremor + Visx mix risks two charting paradigms diverging across PRs.
- shadcn copy-paste model risks component drift.
- **No loading-skeleton spec.**
- **No 44×44px touch-target rule.**

### New requirements added to `requirements.md`

- **NF-A11Y-04** Tap targets MUST be ≥ 44×44px on viewports < 768px and ≥ 32×32px on desktop. Enforced via shared `<Tappable>` wrapper.
- **NF-PRF-07** Cold-cache area pages MUST render skeleton placeholders (Tremor sparkline shells + freshness badge visible) within 200ms; full data within NF-PRF-01 (2s).

### New design notes added to `design.md`

- A `<Chart>` adapter component MUST wrap both Tremor and Visx surfaces. Consumers never import `recharts` or `@visx/*` directly. Enforced in CI by an import-restriction lint rule.
- shadcn components live in a single `components/ui/` folder. Per-feature overrides require explicit code-review signoff and are flagged by a `/* shadcn-override */` magic comment.

### Score

| Surface | Score |
|---|---|
| Spec post §10.7 | 8/10 |
| Current MVP dark theme | 7/10 |

## Part 5 — Error Handling & Edge Cases

### Findings

- Freshness tier system (NF-DAT-06) handles "data is old" gracefully.
- Status page (NF-DAT-08) gives users a way to verify source health.
- **No error-message standard** — multiple failure modes (missing snapshot, sources disagree, low confidence, stale rate) but no consistent UI affordance.
- **No defined behavior** when FRED rate fetch fails.
- **No perf budget enforcement** in CI.

### New requirements added to `requirements.md`

- **F-DATA-01** A standardized `<DataNotice>` component MUST render any of: `missing | stale | sources-disagree | low-confidence | rate-fallback`. Each variant has prescribed copy + icon + action. No ad-hoc "Sorry, an error occurred" strings allowed in user-facing surfaces.
- **F-DATA-02** When the mortgage-rate fetch fails, the affordability widget MUST display the most recent successful rate with an inline `<DataNotice variant="rate-fallback">` showing the `as_of` date and a refresh button.
- **NF-PRF-08** Performance budgets (NF-PRF-01–06) MUST be enforced in CI via Lighthouse CI on every PR. PRs that regress P75 page load > 200ms fail the budget check.

### Score

| Surface | Score |
|---|---|
| Spec | 6/10 |

## Part 6 — Trust, Security & Privacy

### Findings

- Anonymous-first + column encryption + no third-party trackers on financial pages = strong stance.
- **No consent microcopy** for the localStorage write of affordability inputs.
- **No session management UI** ("sign me out of all devices").
- **"Sources disagree" UI** unspecified.

### New requirements added to `requirements.md`

- **NF-SEC-06** Affordability inputs interface MUST display first-party storage microcopy ("We save these on your device only — never sent to our servers unless you create an account") near the inputs.
- **F-SCN-09** Auth settings page MUST include "Sessions" view listing active devices/IPs with revoke-individual and revoke-all actions.
- **F-DATA-03** Numbers where source disagreement (NF-DAT-04) was detected MUST render a `(?)` icon next to the value; clicking opens a popover listing each source's value.

| Surface | Score |
|---|---|
| Spec | 8/10 |

## Part 7 — Product Psychology & Engagement (Built for Mars lens)

### Findings

- Mental model is consistent: every page is `(Area, Snapshot)`. Users learn it once.
- "Show the math" trades initial complexity for trust.
- Risk: density creep as 11 functional pillars land — every page risks becoming too dense.
- **No PWA manifest** — alerts compete with native apps; installable PWA is the cheapest engagement boost.
- **No insights summary digest content design** — a digest that's just "DOM moved 2 days" is forgettable.
- **No language about how paywalled features are signaled** before users invest effort.

### New requirements added to `requirements.md`

- **F-RT-14** Email and in-app digests MUST follow a "3 things to know this week" structure: (1) phase movement, (2) affordability impact, (3) one notable area-level change. Not a metric dump.
- **F-RT-15** A PWA manifest + service worker MUST be shipped in Phase 4 enabling install-to-home-screen + offline cache of last-viewed area pages.
- **F-MON-01** If/when freemium ships (D8), paid features MUST display a small "★" badge inline; tapping the badge explains the tier — no surprise paywall after data entry.
- **NF-UX-01** Default-collapse rule: any page with > 3 metrics in a section MUST default-collapse the 4th onward behind a "Show more" affordance. Rationale: cognitive-load review at every PR boundary instead of just at design time.

### Score

| Surface | Score |
|---|---|
| Spec | 7/10 |
